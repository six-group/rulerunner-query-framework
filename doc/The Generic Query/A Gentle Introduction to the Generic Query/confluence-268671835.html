Keeping your queries

Well, we'll not create a new query here but instead will explain a very useful technique to work with the Generic Query.

If you have executed the above query or looked at the above screenshot, you may have noticed the link that appeared to the right of the Execute button. This link contains all input made in the form encoded as URL parameters. As a consequence, when invoked it will open the form with all input fields already prefilled. There are two uses for this:

  • By dragging the link to your browser bookmarks, you can save the query for later use.
  • By clicking the link, the form will be opened in a new tab with all input already filled in. This allows to continue working on the query while keeping its current version and result for reference.

Especially the second point constitutes a very efficient strategy to create and debug queries: Instead of doing everything in the same form, changing inputs back and forth and, on every execution losing the previous result, you open the query in a new tab to try the modifications.

There is a specific reason why the link cannot be opened in a new tab by the usual right click procedure (clicking it will open a new tab anyway, so there is no need at all for the right click procedure) and why bookmarking the query requires dragging the link. (You need to have your bookmarks bar visible for that, go to the three-dots-menu → Favorites → three-dots-menu → Show favorites bar to enable it.)

The reason is that the link is a so-called bookmarklet, meaning that the target of it is a Javascript snippet. When it is invoked, it takes the host name from the URL of the actual page and adds to it the path and parameters encoded in the bookmarklet to compose a new URL that will then be opened in a new tab. As a result, these bookmarks can be used on any SIX IAM environment but nowhere else.

As a result these links cannot be sent to other people - the receivers will be unable to open them. Instead, send them the URL from the URL bar, which is always updated with the form inputs when executing the query.

Warning!

Keep this in mind when saving your queries:

  • If you bookmark the page on which you executed the query, the bookmark will always take you to the same server, so you cannot use it for different IIQ environments.
  • Bookmarklets don't have an icon, so when you see a Generic Query bookmark with an icon in your browser bookmarks, it's a server specific bookmark.
  • To send queries to other people, copy the URL from the URL bar after executing the query.

Showing the XML

Crafting queries is often an explorative process, especially when confronted with a new object class. Of course, the IIQ object model reference is the definitive source, together with the ObjectConfig that lists all the extended attributes. But a quicker way to identify interesting information is often to look at the object XML. You probably have used the IIQ Debug Pages extensively already. Displaying the XML is very easy with the Generic Query as well, and it offers some important benefits over the Debug Pages:

  • Several objects can be shown on the same page for quick scrolling. This is very handy, especially when complemented with the browser's Search function (Ctrl-F) that will highlight the matches in all the objects in one go.
  • Object selection conditions of any kind can be applied.
  • Object changes can be easily tracked by keeping the current tab with the XML output open and - with a single click - opening a copy of the query in a new tab.

Traversing object relationships

The objects making up an application's data are interconnected. Consequently, the IIQ API allows traversing from one object to related ones by exposing the related objects in object attributes.

As an example, let's add to our entitlement catalog the entitlement owner and its manager. The entitlement's owner (an Identity object) is available in the owner attribute, and the manager of an Identity in its manager attribute (it's again an Identity object). Formatting output

The output of the last query is unsatisfactory in two ways:

  • The expressions don't make up for good column titles. Particularly, they widen the columns unnecessarily, what may be especially painful when a query has many columns.
  • The user ID is a rather cryptic identification of an employee. We might want to represent them in a more human readable way.

While getting additional information about the identities into the output could be done by adding more columns such as owner.firstname, owner.lastname, owner.manager.firstname and so on, in this example we want to keep the two identities in one column each. So we'll use formatting to get a nice representation for them following the pattern firstname lastname (user ID), department.

Entitlement catalog

Field nameInput
TitleEntitlement catalog
Object classManagedAttribute
Filterapplication.name == "IdentityIQ Prod"
Orderingvalue
Outputvalue

manager=owner.manager:format(${firstname} ${lastname} (${name}), ${department})

Note that the closing parentheses behind the ${name} placeholders for the user ID need not be escaped by a backslash to prevent them being interpreted as the closing parentheses of the format instruction, as the templating mechanism by default requires parentheses to pair.

Using collection attributes

Until now, the attributes we used in our queries each contained only a single value. In contrast to that, collection attributes contain collections of values, reflecting containment or to-many relationships.

There are even so-called multivalue attributes that can contain either a single value directly or a list containing multiple values. (An example for this is the attribute value in many classes, other examples are the schema-defined entitlement attributes of Link objects.) The Object Query Framework allows to handle these attributes in a uniform way by transparently converting a single value to a collection when used as one.

Accessing a collection attribute will, instead of a single value or object, return a collection of objects (which is commonly a List), so there are some principal differences in handling them. Let's look what this means.

The Identity class has, among others, the following collection attributes:

  • assignedRoles
  • workgroups
  • capabilities

Let's list some identities with their assigned roles, workgroups and capabilities. The attributes are easy to access but without an explicit conversion, the default conversion would be List.toString(), and we wouldn't be very happy with the output (try it yourself!). To apply an explicit conversion, we have to:

  1. convert each collection element to the representation we want to have for (without this, again toString is used which is often not appropriate)
  2. concatenate the object representations into a string in some suitable way

To convert each element of a collection, creating a new collection with the results, OrionQL's map operation is used. It takes as parameter a conversion, which is an OrionQL expression, and applies it to each element, returning a new collection containing the results. The join operation will concatenate the elements of a collection into a string - without an argument separating them with newlines.

For the workgroups and capabilities we will use the name attribute as representation, but for the assigned roles we want to have the displayName as well, so for it we need to use a format instruction, providing it a template with attribute placeholders that will be substituted for each role:

lastname

The

:map(name):join

type of patterns is a common idiom when dealing with collection attributes in the Generic Query, and you will soon type it automatically. More often than not, the map instruction however will use a format instruction instead of a single attribute. For instance if you are dealing with a collection of AttributeRequests, you might use:

:map(:format(${op} ${name} ${value:join(\n  )})):join

You will soon use this type of constructs equally fluently.

Note the special way the AttributeRequest's value - which is a multivalue attribute (see box above) - is formatted here: The expression value:join(\n ) contained in the placeholder accounts for the fact that value can contain, instead of a scalar value, also a list of these. The expression works like follows:

  • In the case of a single value, using it as a collection (as implied by giving it to the join instruction) will implicitly convert it into a list with it as the only element, and the result will be equivalent to ${value} as there is no other element to concatenate.
  • If the value is actually a list, the elements will be concatenated by the string "\n  ", causing them, starting from the second element, to be on a new line each, indented with two spaces. The placeholder ${value} would create a standard list representation of the form [element, element...] on a single line instead.

Resolving objects

Not all object relationships can be directly queried using the IIQ API. In most cases, the reason for this is that intentionally no reference constraint was implemented to achieve a decoupling. You wouldn't want, for instance, audit records prevent deleting the objects they reference, nor deleting these objects cause deletion of the audit records.

To follow these weak references, they can be resolved by using an OrionQL lookup instruction. The easiest case is a query by name or ID, so as an example let's list some roles together with their role_manager (the attribute contains an Identity ID), representing it by the user ID:

Retrieving additional data

Even more than for to-one object relationships, only a fraction of the virtually unlimited number of to-many relationships between objects are exposed as object attributes in the IIQ API. Imagine, for instance, the list of AuditEvents written during the processing of an IdentityRequest. The query possibilities are virtually endless.

However, like for resolving weak object references (see the preceding example), by using OrionQL's lookup instruction it's very easy to get access to object collections not directly reachable by API methods. The lookup instruction performs a database query and returns the result as a collection.

You may notice at this point that this way, there is no principal difference between object relationships directly accessible through methods of the IIQ API and object relationships requiring a database access for resolution: The lookup instruction makes them equal, working just like a collection attribute (which lets Hibernate do the database query for you behind the scenes).

As an example let's list managers with their subordinates. We'll represent the subordinates in the following format: firstname lastname (user ID), department. Besides explicitly supplying a column title, for better understanding the expression has been formatted into multiple lines using indentation, taking advantage of the Rulerunner's hidden linebreak feature. The :map(:format(...)):join pattern is the same that you already saw in the collection attributes example above:

Dissecting strings

Let's return to our entitlement catalog query above where we showed the entitlement owners and their managers. Instead of the owner's manager we now want to show the owner deputies which are stored as a comma separated list of user IDs in the deputy_owner attribute of the entitlement.

In addition to resolving the weak references the user IDs constitute, this requires us to first extract them from the comma separated string. Regular expressions come to help: We use the extract instruction to get all matches of a regular expression as a list. A simple pattern, matching any nonempty sequence of characters that are not commas or spaces, is sufficient for the purpose.

Having this list, we can apply what we already know about processing collections using map and join. This time however, the map instruction has more work to do since the input it is fed is not a collection of objects but a collection of strings that first need to be turned into objects using a lookup instruction. So, the map instruction contains a sequence of processing steps - resolving the object and formatting it. For better readability we again format the column expression into multiple lines with indentation, using the Rulerunner's hidden line breaks feature:

Using selection rules

Selection rules permit to check conditions on objects that are difficult or impossible to check in the database. When a rule set consisting of one or more CerberusLogic rules is provided in this field, the iterated upon objects will each be checked against it. This way, additional selection conditions can be applied.

For performant queries, always try to narrow down the selection as far as possible using a Filter and leave only the remaining selection tasks to the Selection rules. The filter will be evaluated already in the database, preventing nonmatching objects to be fetched at all.

If possible, always prefer to put the selection condition into the rule header, since it will be checked before synthetic attributes and collection expressions have been computed.

Let's find out if there are entitlements with owner deputies from a different department than the owner. For this, we extend our last query with a selection rule that checks, for each entitlement, if there is any deputy whose department is not equal to that of the entitlement owner.

To check the deputies, we copied the part of the deputies column expression that extracts the user IDs and converts them to Identity objects into the any selector. Removing the hidden line breaks gave us better readability for the CerberusLogic syntax, sacrificing the readability of the OrionQL expression.

For each deputy Identity in the list we check if the department is not equal to the entitlement owner's department. To access the latter we need a reference - normal values in this place would be interpreted as literals. The reference first navigates back to the entitlement using a parent reference (^) - since we computed the list of deputies from the entitlement, this is the parent for all the list elements - and form the entitlement to the owner, finally accessing the department:

Using synthetic attributes

The last query is a good candidate to show the use of Synthetic attributes. In essence, synthetic attributes are variables you can define for the results of computations that are expressed as either OrionQL expressions or CerberusLogic rules. The variables will be accessible in the same way as as normal object attributes, masking attributes with the same name, if any. Hence the term synthetic attributes.

The above query suffers from containing the same computation twice: The list of owner deputies is needed for the object selection (Selection rules) as well as for listing the deputies in the deputies column (Output field). While this does not cause a big performance penalty thanks to the fact that the results of database queries are cached, and only the relatively fast in-memory operations are executed twice, having the same expression twice makes the query more complex than it needs to be. So, let's factor out the common part into a synthetic attribute.

Due to the fact that synthetic attributes are computed only after the object has passed the rule header checks, we need to move the selection condition from the rule header to the accept clause now.

Unfolding collection expressions

In this section, we'll cover an extension for linear queries that is a concept important enough to justify a section on its own: Unfolding master-detail relationships.

We encountered master-detail relationships already when working with collection attributes and OrionQL collection expressions: They represent, for an object, multiple related objects (often called child objects), and outputting them in a column required us to first apply some mapping operation to the collection and then concatenating the result of it into a string - commonly creating a multiline cell. The data representation we got this way in the queries is fundamentally different from how master-detail data is commonly queried in SQL (using a join) or represented in an Excel table that both dedicate a row on its own to each detail object, replicating the master object in all those rows.

Moving a collection expression from the Output field (where it is part of a column expression) to the Collection expression field causes the Generic Query to unfold the child collections into separate rows. Technically, the collections returned by the collection expression for every object will be iterated upon, feeding their elements into the remaining query processing in place of the original object. In other words, the query result will be made up of, instead of the primary objects, all their child objects (secondary objects). As a result, the primary objects will not be part of the query result any more. They will be available, however, from each of the secondary objects through parent references.

Unfolding child collections allows, besides creating join-like output and applying selection rules on the child object level, to also handle some challenges the IIQ object model offers. When we'll come to grouping in the next section, you will see even more uses for it.

Grouping queries

In the linear queries we examined up to here, each object satisfying the selection conditions was written to the output table immediately, row by row, in the sequence of the object iteration. Instead of creating linear tabular output like this, grouping queries collect the objects according to the specified grouping criteria, and after processing all objects convert the collected result to an output representation.

Let's incrementally get familiar with what opportunities this opens.

About sorting the result set

In most cases when doing grouping queries, there is no reason to specify an Ordering to sort the objects in the database since the final output will anyway be determined by the ordering of the grouping key values. Apart from causing unnecessary work in the database in this case, sorting may even be counterproductive, since it will remove the small amount of randomness you have in your data set when for a quicker response you limit its size by using Maximum rows while prototyping a new query. There is, however, one exclusion from this rule: When objects are not counted only but grouped, and the order in which they appear in their buckets matters, an Ordering should be used.

An alternative that helps to avoid the database sorting of the entire result set is is to use Groups postprocessing to sort the buckets.

Categorizing objects

Let's do another grouping to show how objects can be categorized using more complex computations that we have used so far. This time we want to get an overview about the sizes of access requests we have, measured in the number of roles ordered.

Grouping the IdentityRequest objects by the exact number of roles ordered would, however, fragment the result, preventing it from providing much insight. What we therefore need is to create some size categories which we can use for the grouping. We will use an OrionQL attribute computation rule for this since it allows value computations depending on condition checks.

Apart from this, the query demonstrates a couple of other noteworthy techniques not demonstrated so far:

  • The number of roles ordered in the access request is computed as a synthetic attribute numRoles since it needs to be examined several times in the attribute computation rule for the requestSize synthetic attribute. The Generic Query will execute the two blocks of attribute computations in separate passes, so that the second block has access to the synthetic attributes computed by the first one.
  • The computation of numRoles is done by counting the identity request items by application and operation, followed by accessing the count for application IIQ and operation Add with two chained attribute accessors that retrieve the "attribute" IIQ followed by Add from the two-level mapping produced by count.
  • Since the IdentityReqest's operation attribute has an Enum type, it needs to be formatted as string to be accessible in the mapping using a string.
  • IdentityRequests containing only role removals will not produce a value for Add, having the corresponding attribute accessor return null. An expression alternative with the literal value 0 is appended to the expression that supplies the value for the numRoles synthetic attribute in this case. (It would not be set otherwise, causing an error when accessing it later, as the IdentityRequest class does not have an attribute attributes containing an attributes map holding attributes not defined by the class.)
  • The labels used for the size categories have been chosen carefully to alphabetically sort in the desired sequence. (The rows and columns of the matrix are sorted in ascending key order.)
  • To collapse all departments of each business unit into a single row, we replace in the target identity's department attribute everything that follows the first minus (-) with an asterisk (*). (Alternatively, we could have used match to extract the part up to the first minus, giving a slightly different result, but we have used match already in an earlier example.)

By sorting the identity requests by descending creation timestamp and limiting their number to 1000, we select the latest 1000 access requests. (We'll meet this number again in the output - in the total totals cell at the bottom right of the matrix.)

Unfolding maps

The above technique - showing data in a two-dimensional grid layout - is not limited to grouping objects fetched by a query but has a more general nature that we will illustrate with another example that does something unexpected: We'll present the data contained in a stream of Maps in a tabular view by taking the maps apart and re-arranging the key-value pairs into rows and columns. While the maps can be read from any origin - even computed by a grouping OrionQL expression - let's use a simple example here: attribute maps.

We chose attribute maps for this demonstration because these maps are available out of the box. But remember, that you can mangle this way any map you get hold of or produce yourself.

In the very beginning of this guide, when displaying object XML, it was mentioned that maps, in particular, attribute maps, can be formatted using the pprint instruction. Dealing with null values and Enums

When using grouping - either via the Grouping expressions field or using OrionQL grouping instructions - there are two situations where you need to take special care about what your OrionQL expressions return:

  • If grouping keys are to be sorted (as the Generic Query always does when Grouping expressions are used, and OrionQL's grouping instructions do when asked to do so), null values are not permitted. To cope with null values that may appear in the data, either a format instruction or an expression alternative can be used:

    Examples
    :format(${type})
:format(${type,n/a})
type|"n/a"

  • If OrionQL grouping instructions use expressions that return Enums, they will not be indexable using strings. To ensure indexing will be possible, convert the Enums into strings using a format instruction:

    Example
    items:count(application#:format(${operation})#value).IIQ.Add:map(key)

    This expression, applied to an IdentityRequest, gets a list of the roles that were ordered by counting the request items by application, operation and value and then getting all keys from the bucket for application IIQ and operation Add. The operation value is an Enum, and using it directly would .Add to always return null. (By counting also by value and then re-extracting the values from the bucket's keys, instead of directly grouping the values, this expression can handle multivalue request items and role duplicates - although we probably don't need to expect them.)

Escaping

Escaping is used to change the meaning of characters by prepending them with a special character which is often (and particularly in the context of the Generic Query) the backslash (\). Often, but not exclusively, escaping is caused by embedding one language into another, for instance regular expressions into OrionQL or OrionQL into CerberusLogic. Consequently, there are several situations where you will be confronted with escaping when using the Generic Query for non-trivial stuff, and you will commonly get strange errors when you don't get it right. The good message is: Most often you will find the reason by carefully looking at the error output to see where the parsing went havoc. As a general orientation, here are the basic rules to get it right from the beginning:

  • OrionQL constructs that accept non-OrionQL text (most prominently match/extract/replace and cache instructions), require escaping for all characters that would otherwise be considered part of the instruction's syntax. Watch out for unescaped commas, closing parentheses or backslashes if you have problems.
    Note that the format, lookup and select instructions try to parse the embedded constructs syntax-aware and thus in most cases don't require escaping.
  • When using an OrionQL expression as data source that uses dotted attribute syntax, be sure to escape the dots when accessing these attributes to have the dot recognized as part of the attribute name.
  • When embedding OrionQL expressions into CerberusLogic code (they have to be represented as string literals there), any backslash or double quote (") needs to be escaped.

While unnecessary escaping won't harm from a functional perspective, so you can always use it in case of problems if unsure, you should always remove it after finishing debugging to not cause unnecessary complexity.

Remember: Whenever you are confronted with a strange error, watch out for a missing escaping!

Date filters

We used this already in the example about analyzing logs, but let's, without providing another example, get into more detail about specifying Filter conditions involving Date values. As stated, what is specified in the the Filter field is in fact an OrionQL template, so placeholders can be used to format into it the results of expression evaluations. What you will commonly use is the quoting placeholder:

$"expression"

It will convert the result of evaluating expression into the representation expected by the IIQ FilterCompiler:

  • Strings will be quoted (e.g. "FunctionalRole")
  • Dates will be converted into a string of the form DATE$milliseconds (e.g. DATE$1683699887569)
  • Lists will be converted into a list literal of the form {value, value, ...} (e.g. { "ApplicationRole", "FunctionalRole" })

Particularly interesting in the Filter is using the OrionQL date instruction and especially its now variant in a placeholder to specify Date conditions. The difference between date and now is that now uses the current system time as input, whereas date is used for Date literals or using/converting Dates given to it as input. Based on that, the following options for manipulating Date values are available:

  • Specifying a Date literal by using date with a parameter in the format yyyy-MM-ddTHH:mm:ss.SSS with any precision (omitting trailing parts is equivalent to truncating at this time unit). Examples:
    • $":date(2023)"
    • $":date(2023-05-10)"
    • $":date(2023-05-10T13:19)"
    • $":date(2023-05-10T13:19:17)"
  • Adding or subtracting a timespan specified in days, hours, minutes and/or seconds. Examples:
    • $":now-14d"
    • $":now-6h30m"
    • $"created+10s"
  • Truncating and applying an offset to a Date value: Specifying as the instruction parameter a signed offset in either years, months, weeks, days, hours, minutes or seconds will truncate the input value to the specified unit, and after that apply the offset by using correct calendar calculations. (Particularly, specifying an offset of 0 returns the start of the current year, month, day, hour etc..) The units are Y, M, W or w depending on the week's start day, d, h, m and s. Examples:
    • $":now(-0Y)"
    • $"created:date(+1d)"

Using this technique allows, without doing manual time calculations, to easily write filters that select objects, for instance by created timestamp. The following Filter will select objects that were created within the last 10 minutes:

created > $":now-10m"

Limiting column width

The Generic Query does never reflow text, so output will always be shown as produced. As a result, in the case of long strings such as error messages, columns may become uncomfortably wide, also wasting screen space in cells not filling the width. When the value causing the column to be widened is a list (a multivalue attribute, for instance), the problem can be easily addressed by folding it into multiple lines like, for instance:

memberOf:join(\n  )

The explicit parameter given in this example demonstrates how the follow-up lines can be indented if needed.

This technique doesn't help when what is widening the column is a single value. In this case, regular expressions can be used to mangle the string. There are several options:

  • Limiting the overall length by matching only up to a maximum number of characters from the start using the OrionQL match instruction. This is rough though.
  • Replacing the part following a certain number of characters by three dots using the OrionQL replace instruction. This gives a particularly nice output, indicating that the text has been truncated.
  • Breaking the value into multiple lines by inserting newline characters after every certain number of characters using the OrionQL replace instruction. This doesn't give a particularly nice output but may better suit the purpose than simply cutting off text. If needed, the newline characters can be amended with marker characters or indentation of the continuation line to signal the inserted line break.

We will look at the two replace options since they are the preferable ways of addressing the problem.

Since Java regular expressions, as used by OrionQL, have the dot wildcard (.) work in multiline mode by default, the replace instruction will handle multiline strings without additional effort.

For counting the characters we will use the curly braces quantifier. It allows to specify the number of characters directly.

To replace the text exceeding a certain limit of characters by three dots, we use a capture to match the text to keep and replace the whole match by the capture and three dots. To be sure we do the replacement only if what follows is more than the three dots we plan to add to the kept text, we require what comes after the capture to be at least four characters. Let's limit the length to 80 characters, meaning we have to cut off after 77 characters so we can add three dots after them. This is the regular expression to match (we will require there to be more than three characters to replace them by three dots, of course):

(.{77}).{4,}

It matches 81 characters and returns the first 77 of them as capture. The capture will be referenced using $1, and three dots will be added to form our replacement:

:replace((.{77}\).{4\,},$1...)

Note the escaping for the closing parentheses and the commas contained in the regular expression.

When the expression does not match, what is the case if there are only up to 80 characters, it will be returned unchanged. The following query demonstrates the technique:

To split long strings into multiple lines, we match the desired number of characters, capturing them, and replace them with the capture and a newline. Let's limit the line length to 80:

:replace((.{80}\),$1\n)

If you apply this to a description field, the generated output is not particularly readable since words are broken by linefeeds. To overcome this, we can extend the match to also include any non-blank characters following the maximum-length matched so far. This will cause linefeeds to be inserted only before a blank character:

:replace((.{80}[^ ]*\),$1\n)

As a side effect, there is now no more hard limitation for the output line length - the number given in the quantifier instead specifies the minimum length of a line before a possible split. Modifying the regular expression to impose a hard limit on the output line length by is left as an exercise to the reader. (Hint: Limit the allowed number of additional characters.)

Where you want to allow this kind of soft split to happen, of course, depends on the type of text. Blank characters are a good choice for natural language text, but for instance Distinguished Names you would probably want to split after a comma.

Abbreviating fields

When trying to shorten representations to squeeze as much information as possible into the available output space, it might be not an option to simply cut off strings after a certain length as described in the previous section. Instead, some intelligent abbreviation might need to be computed. A good example is the source field in Audit records that can contain values like these:

  • Identity Refresh
  • Account Aggregation
  • Provisioning Workflow
  • wfl: SIX LCM AccessRequest Control Split
  • wfl: SIX LCM AccessRequest Execution Control New Split
  • Source Mapping Rule - SIX IdentityAttribute npaOwner

Suppose you want to put into the output of your query an "audit fingerprint" that is made up from application, attribute name, attribute value (the latter one already shortened using the approach from the previous section) and the source of the audit event. Then, a formatting template like

[${application}/${attributeName}] ${attributeValue:replace((.{57}\).{4\,},$1...)} (${source})

would produce audit fingerprints like this:

[SAP/Roles] CBICLNT001\ZBW_GHRPA_XX_MANG_CR (Account Aggregation)

If you have such audit fingerprints in multiple columns, like in this query, shortening the source would be interesting, making the output less wide and thus easier to navigate. To come up with a useful abbreviation, you can use the replace instruction to perform transformations like the following (using the source attribute as an example):

  • Remove everything from the string that is not a capital letter (use this with care!):

    source:replace([^A-Z],)

    This will give you, for instance, AA for Account Aggregation or IR for Identity Refresh. This type of abbreviation can, however, not only be a bit cryptic to decode in certain cases but it can even completely remove all information if there are no capital letters, so here is another option:

  • Concatenate into a CamelCase abbreviation each capital letter and up to two lowercase letters following it:

    source:replace(([A-Z][a-z]{0\,2}\)[^A-Z]*,$1)

    This will give you, for instance, SouMapRulSIXIdeAttOwn for Source Mapping Rule - SIX IdentityAttribute npaOwner - instead of SMRSIXIAO like the first variant would give.

There are, of course, much more ways to customize the abbreviation generation, but you got the idea.

Inline aggregation

We have used this technique already, but it's important enough to get its own section. Besides the Generic Query doing aggregations using Grouping expressions, OrionQL also provides aggregation capabilities (count, group and index instructions). They can be used everywhere an expression is accepted, particularly in column expressions, and we will show a non-trivial example of using inline aggregation here.

Let's list the access request for some identity, listing in two columns the approved and rejected roles. For this we will examine the approvalSummaries collection attribute. It contains a list of the approvals.

The logic behind the query is to first count, separately for each access request operation (Add, Remove, Retain), for each role the approval decisions (Finished, Rejected, Canceled). Then for each role, if there is a Rejected decision, the operation for the role was rejected, otherwise it was accepted. (To be more exact, we would have to also check in this case if there was a Finished, but let's omit this for simplicity now.) As there may be different operations requested regarding a role, we use a top level grouping to keep them apart.

To ease understanding the data transformations this query performs, the output contains the raw aggregated data in an additional column. This also demonstrates how queries can be built up incrementally when developing them. In the final version of the query, of course, this column would be removed. For the discussion below, let's provide the aggregated approval data for one of the access requests:

Looking at the data, we see that the access request was about ordering 6 roles of which 5 were approved and one (Role705) rejected, and there was nothing other in the access request than these adds. Let's now look at how this aggregation is done.

To get at the IdentityRequest's approval decisions for the roles, a nested iteration is needed: The approvalSummaries attribute contains a list of approval summaries, each of which contains in the items attribute of its approvalSet a list of approval items. The latter are what contains the operation and the value (role name), while the approval summary carries the decision.

The nested iteration is done using the OrionQLs flatten instruction which delivers the nested iteration's leaf elements (the approval items) as a flat list. The elements from the outer iterations (here the approval summaries) are accessible from the elements of this list via parent references. To perform the nested iteration, all the flatten instruction needs to be given is, for each iteration level, the path from an object to the collection for the next iteration level:

:flatten(approvalSummaries#approvalSet.items)

The flatten instruction is a very powerful construct - it allows to iterate a nested object hierarchy in one go - by only providing a virtual path to the leaf elements, having hash marks (#) for the iterations on the way down. You'll probably need some time and practice to master it, but once you do, it will let you do complex stuff very easily.

The result of flatten is then fed into the count operation to do the aggregation, resulting in the nested map shown above:

:count(:format(${operation})#value#:format(${^.state}))

Note how the Enums for operation and state are converted into strings using format, to permit the buckets they produce to be accessed using string keys.

The aggregation result is stored in a synthetic attribute approvals to have it available for multiple columns without recomputing it for each of them.

To get the list of approved or denied roles, in each column expression we first extract from the synthetic attribute approvals the bucket for the respective operation (Add, Remove, Retain):

Sorting pprint output

Until now, we didn't care about the pprint output being unsorted. The reason for this is that OrionQL's grouping constructs by default use HashMap, so the keys will be in hash order, effectively looking unsorted. If sorted output is desired, the grouping constructs can be directed to use a Map implementation with sorted keys (TreeMap) by specifying the > flag:

items summary=items:count(>operation#application):pprint

When the Map submitted to pprint was created by IIQ, such as the attributes mapping of an IIQ object, a workaround can be applied to sort the top level keys by re-indexing the mapping:

attributes:index(key,value):pprint

This will output the attributes in sorted order.

Prefetching data

When using OrionQL lookup instructions, the database query results are cached. Repeating the same lookup for multiple rows thus does not cause the query to be executed multiple times. However, there may be a performance issue when doing many small database queries in tight sequence. A better approach is often to fetch all data that will be needed upfront with one query, storing it in memory in a lookup table and using that for the lookups. Doing one big query instead of turning to the database with a separate query for each lookup might result in a substantial speed gain.

While OrionQL permits application code to provide lookup tables to have them available for use in lookup instructions, this approach is not feasible for the Generic Query. Instead, OrionQL constructs can be used to create lookup tables on the fly, as we will see in the next example. The example is also educating by exacting some other challenges.

Let's count, for each EAM application, how many firewall and non-firewall entitlements from automatic endpoints are built into its roles. A firewall entitlement is identified as starting with CN=g-firewall-.

We solve the task by reading the bundle-entitlement connections stored in BundleProfileRelation objects and counting them by the entitlement's category (firewall or non-firewall) and the bundle's EAM application. To select only entitlements from automatic endpoints, in the Filter we use an in condition on sourceApplication.name, preparing the list of all automatic endpoints inline by reading the names of all Application objects representing automatic endpoints using this selection condition:

!type.in({"DelimitedFile", "ICCLocalManagedApplication", "SixLocalManagedApplication", "Logical" }) &&
authoritative == false

To exclude entitlement roles from the selection, we request the required attribute of the BundleProfileRelation to be true. This will select only the bundle-entitlement connections resulting from role requirements.

The bundle's EAM application is determined from the bundleId by a database lookup. Note that while SIX_EAM_APPLICATION is defined in the ObjectConfig as a named column, fetching it directly would yield an error (Could not find class for join property SIX_EAM_APPLICATION in packages [sailpoint.object]). We work around this by getting it from the attributes mapping, what, in performance terms, is not free however:

bundleId:Bundle({attributes}).SIX_EAM_APPLICATION

To cope with null values, the attribute is then converted to a string using a format instruction. Unfortunately, this ruins the numerical sorting that we could enjoy otherwise, so if the long type of the attribute wouldn't prevent us to do so, we would rather provide a default value instead by using an expression alternative.

Getting at the bundle's SIX_EAM_APPLICATION the way shown above lets the query turn to the database for each role referenced by any of the entitlements individually. Let's now get to the topic of this section and use an on-the-fly lookup table that is filled from a single big query on first use to get rid of the about 9000 separate database queries. For this, we change the above expression to the following:

:cache(bundles,\
  :Bundle(>{id,attributes}!type.in({ "EntitlementRole", "OrganizationalRole" }))\
  :index(id,attributes.SIX_EAM_APPLICATION)\
):get(bundleId)\

The lookup table is computed inside the cache instruction by reading, for all bundles with appropriate type, the id and attributes and building from this an index that maps each id to the SIX_EAM_APPLICATION of that role. By wrapping this expression into a cache instruction we ensure that it is computed only upon first invocation and then stored in the cache under the specified key bundles, so that from then on it will be taken directly from the cache. The lookup table then will be used by a get instruction to retrieve the SIX_EAM_APPLICATION for the role referenced by the current BundleProfileRelation, supplying bundleId as the lookup key.

Assessing the performance improvements

When testing the performance of the caching and non-caching variants of our query, there is no clear winner. One reason for this is that the query is spending a big part of its time reading the about 20000 BundleProfileRelation objects. Any variation of the time needed for this will mask the differences caused by the different EAM application lookup implementations. However, what we can state at this point is that the in-memory lookup at least didn't result in a substantial performance gain.

It's important to note that besides creating additional complexity, this pattern may even hurt performance when applied carelessly. This can easily be the case when it causes much more data to be fetched from the database than will be effectively used. In such a case it will be more advantageous to use the simple load-as-needed approach. In other cases however, it might even make the difference between the query being usable or not.

As a general guidance, keep the tool in your arsenal but use it consciously.

The gain or loss resulting from an inline lookup table depends on the ratio between the effectively used and all prefetched data as well as on how expensive it is to load each of these data items: If only scalar attributes are needed and can be fetched from database columns directly (possibly also navigating an object relationship via a join), the expenses are low. If a CLOB, like in our example the attributes mapping, needs to be fetched and deserialized, the cost is significantly higher, and if even the objects have to be built, there might be no gain from batching the database accesses at all. If you are unsure about the value a lookup table provides, do some measurements. The following queries load, for three different types of data, the bundle data either directly in batch or by looping through all bundle IDs:

  • scalar attribute (type): scalar batch (2.1 s) vs. scalar loop (21.1 s)
  • joined scalar attribute (owner.name, note the escaped dot when referencing the attribute): join batch (2.4 s) vs. join loop (21.6)
  • complex attribute (attributes): map batch (16.3 s) vs. map loop (> 40 s)
  • object: object batch (29.5 s) vs. object loop (36.6 s)

The query execution times often vary considerably (as can be seen by the map loop taking more time than the object loop), so for accurate conclusions, the measurements should be repeated a couple of times.

As we are at it, let's also measure the time needed by the primary query - for the BundleProfileRelation data. The query from this section loads the objects, so we can expect a speedup when fetching only the two attributes that are needed. The following queries omit the role attribute stuff, so only the primary query is measured:

  • loading attributes: BundleProfileRelation attributes (1.6 s)
  • loading objects: BundleProfileRelation objects (15.6 s)

As is commonly the case, omitting the object loading gives a speedup by factor 10. When changing the query from this section in this way, it speeds up considerably, and the performance gain achieved by the lookup table approach becomes more noticeable:

About the cache key

The cache key is a formatting template in which placeholders can be used to distinguish the results of computations differing only in their inputs. Since in the case at hand the computation uses no input but has the selection condition hardcoded, no such distinction needs to be made, and the cache key is a simple string.

In contrast, imagine caching for instance entitlement data, where instead of reading all entitlements upfront, possibly wasting time reading entitlements for large endpoints that will never be referenced later, you want to read only some subset to cover the immediate need (covering a single endpoint, for instance). In such a case, you will have a selection condition that depends on the input and potentially create multiple lookup tables that of course need to be kept apart in the cache by having the keys reflect the different selection conditions (for instance, containing the endpoint name if the selection is by endpoint). With such a setup, the entitlements of any endpoint will be read only when (and if) the first entitlement of the endpoint is referenced.

As an example, let's build a lookup table for entitlement owners to use when listing role entitlements. As a drive-by exercise for using flatten we'll read the entitlements from the role profiles instead of the new BundleProfileRelation table available since IIQ 8.3. The following Collection expression will do the job:

:flatten(flattenedRequirements#profiles#constraints#value)

Starting from the role, this expression iterates over all required roles (also following any requirement chains by using the special API method getFlattenedRequirements()), for each of them iterating over its profiles, for each of them iterating over its constraints, and for each of them iterating over all values. This thoroughly follows the one-to-many relationships codified in the IIQ object model without regard to how we actually use it (we have only one profile per entitlement role, one constraint per profile and one value per constraint), while the expression builds on that the constraints are all of type containsAll).

The output of this expression is a stream of values (entitlement names), having each attached via a parent chain of constraint → profile → entitlement role → role the context information needed to identify the entitlement - namely the entitlement type (attribute name, on the constraint) and the endpoint (application name, on the profile). This means (by going the flatten path backwards), when iterating over flatten's output:

  • @ refers to the value (entitlement name, a String), this is the iteration's current object (the this object)
  • ^ refers to the constraint (a Filter).
    Alternatively, the more self-documenting expression ^Filter can be used.
  • ^.^ refers to the profile (a Profile).
    Alternatively, the more self-documenting expression ^Profile can be used.
  • ^.^.^ refers to the entitlement role (a Bundle).
    Alternatively, the more self-documenting expression ^Bundle can be used.
  • ^.^.^.^ refers to the role (a Bundle).
    Alternatively, the more self-documenting expression ^^Bundle can be used.

Now that we have the triplet entitlement key that allow us to lookup the ManagedAttribute for determining the entitlement owner, the lookup table comes in: To not query for each entitlement, pulled out of a profile, the entitlement owner with a separate database lookup, we fill a lookup table on first use with one big query and get the entitlement owner from it. Furthermore, to not load entitlements that will possibly never be referenced by the query, we will create the lookup tables per endpoint as a two-level (attribute → value) lookup table, caching it by an endpoint specific cache key entitlementOwners_endpoint. Consequently, the database query as well as the cache key template reference application.name from the profile (^.^). This expression returns the lookup table for the endpoint of the current entitlement:

:cache(entitlementOwners_${^.^.application.name},\
  :ManagedAttribute(>{attribute,value,owner.name}application.name == $"^.^.application.name")\
  :index(attribute#value,owner\.name)\
)

The entitlement owner will be retrieved from it using this two-level lookup by entitlement attribute name and value:

:get(^.property#@)

The final query lists the selected roles with their entitlements including the entitlement owner. We already dissected objects by iterating over the elements of a collection attribute using the Collection expression field. We also used the OrionQL flatten instruction in column expressions. Free iteration

The primary function of the OrionQL flatten instruction is walking an object tree given by a generalized path to the objects to be iterated upon, that is made up of linear segments interspersed with iterations. We have seen these kinds of uses in the preceding examples.

The flatten instruction is, however, more general - it does nested iterations - so we can pull even more power out of it. The collections iterated upon on each next lower level are specified using expressions that are applied to the elements of each iteration level. In the normal case of these expressions returning collections related to this object, the result is walking an object tree. However, the expressions are free to return any collection they want - bound to the object given to it or not. We call the latter case free iteration, and it effectively walks a matrix instead of a tree. Consider the following instruction:

:flatten(:Identity(>active == true)#:Bundle(>disabled == false))

It will iterate over all active identities, and for each Identity, it will iterate over all bundles that are not disabled. Both expressions entirely disregard their input by having the selection conditions encoded within them literally. As a result, a huge matrix is walked by multiplying both sets - similar to an SQL table join without a join condition.

Note that the matrix is never created physically in memory, as the flatten instruction creates only a stream of objects.

Let's apply this technique to our problem from the previous section to produce the full matrix of all role types and role attributes. For this, coming from the ObjectConfig Bundle, we first iterate over all role attributes and then over all role types (we could do it the opposite way also). Reading from OrionQL expressions

Let's explicitly discuss a powerful technique that we already used several times in the extension boxes above: Changing the query's input source to Expression.

The normal way queries are done is by entering an IIQ object class name into the Object class field. This causes the objects of this class, as selected by the Filter condition, to be fetched from the database and processed. There are two reasons to deviate from this schema:

  • Performance. As long as the Generic Query is unable to directly load only the attributes needed by the query but always loads the complete objects (effectively disregarding the Load objects flag), certain queries may be prohibitively slow. As a workaround, in the meantime the Generic Query can be directed to read its input from an OrionQL expression since OrionQL allows to do attribute-only queries by explicitly listing the attributes to fetch.
  • Complex data processing. While the Generic query provides basic features for object grouping and related output layout, it doesn't provide extensive data mangling capabilities on its own. When this is needed, the powerful data aggregation and transformation capabilities of OrionQL can be utilized by directing the Generic Query to read its input from an OrionQL expression, so the Generic Query is used only for output rendering.

As we had an example for the first case already, we'll now provide an example for the second.

When we introduced the Matrix view, we displayed role assignments in a grid layout with the identities in the columns and the roles in the rows, and the cells showing the assignments. While this is already quite nice, it quickly reveals that there are roles assigned to the same sets of identities. It would be interesting to merge them into a single row to provide even more insight about the role assignments. Let's do this.

What we obviously need to do is grouping the roles by the sets of identities that have each role assigned. This will collect all roles that are assigned to the same set of identities into one group, so we can render the assignments for all these roles in the grid as a single row.

It's clear at this point that we need two consecutive groupings: First, we have to group the role assignments by role to get, for each role the set of identities that have this role assigned. In a second step we then need to group the output of this the opposite way - grouping the roles by the sets of assigned identities. The Generic Query doesn't allow us a processing like this, since it provides a single grouping step only, the result of which is then immediately converted into output.

It is, however, no problem to do such processings in OrionQL - the pipeline-like notation it provides fits the problem naturally. The result can then be fed into the Generic Query's output processing for rendering by having it read its input from the expression.

The first part of the processing pipeline is quite straightforward: For all identities in scope we read the names of their assigned roles and count, for each role, how often it is assigned to each identity:

:Identity(>{name,assignedRoles.name}\
  department == "DEP001" &&\
  assignedRoles.name.notNull()\
)\
:count(assignedRoles\.name#name)

Note that we need to explicitly suppress the null role added by the outer join behind the scenes for identities with no assigned role.

Suppressing duplicates

You might wonder why, being interested effectively in a list of identities per role, we count the assignments by role and identity instead of directly grouping the identities by role. The reason for this detour, that requires us to later extract the identities from the keys of the inner mapping, is that it removes duplicates.

The mapping this produces is essentially equivalent to what we already produced earlier in this guide when introducing the Matrix view. The only difference is that here we are using the role and identity names whereas earlier we created a more readable output using format. (We will elaborate on that later, but for the beginning let's keep the example as simple as possible.)

Using pprint, we can check the result we have so far:

Note that by default the OrionQL groupings do not sort the map keys. We will have to keep this in mind when using the result.

After we have this grouping ready, the interesting part follows: In the second pipeline stage, we re-aggregate the mapping we have produced so far the opposite way around - grouping the keys by their values. This will group the roles (which are in the keys) by the lists of identities they are assigned to (which are in the values). This is easily done by appending another group instruction to the expression.

As you already know, a Map represents a collection of key-value pairs, providing keyed access for value retrieval. So, feeding a Map into an instruction that expects a collection as input - like the group instruction - will have that instruction operate on the map's key-value pairs. To group the keys from the key-value pairs of the above mapping by value, we basically do the following:

:group(value,key)

This groups the Map's entries by value, storing in the output only the key instead of the key-value pair itself.

Since the value, as we produced it in the preceding aggregation, is not quite yet the list of identities we want to use as the grouping key but a Map of role assignment counts (we took this detour to remove duplicates), we need to extract from the Map in the value the identities, which are in the keys. Again, we operate on the Map's collection of key-value pairs, mapping them to their keys, what gets us the list of identities. Since in the first aggregation we didn't request the keys to be sorted, we'll get the identities in random (hash) order. This shouldn't be a problem since maps containing the same set of keys should return them in the same order but to not rely on that, we sort the list of identities to make sure there cannot be differences by order only hurting our grouping. This is our second pipeline stage:

:group(value:map(key):sort,key)

As we can see by checking the output of the complete expression so far, the mapping it produces maps the lists of identities to the lists of roles that are assigned to each identity:

To get this data into a Matrix representation similar to the one we had earlier but with one row per role group, we need to take the identity lists in the keys apart again. Without that we would get a column for each [unique] list of identities, what would rather obscure the picture than providing additional insight. So, instead of simply iterating over the above Map's entries to do the output, we apply a flatten operation to it to do a nested iteration - the outer one over the Map's entries - the groups of roles with identical assignments - and the inner one over the elements of each entry's key - the list of identities. Since the outer iteration is on the flatten's input object itself (which is the Map we produced above) as opposed to a collection to be reached from it, the path for the first iteration is empty, letting the argument of the flatten instruction start with a hash sign (#):

:flatten(#key)

This instruction will turn the map we produced into a sequence of the identities contained in its keys, with each of them having the key-value pair it was taken from as the parent.

The Generic Query as a layout engine

Making the Generic Query read its input from an expression like we did here effectively reduces it to a layout engine for result rendering, while all the processing is done within the expression. This is very important, since it means that you can do any processing, without being limited to what the Generic Query's input form seems to offer.

That the Generic Query is, especially what concerns its grouping feature, in fact a layout engine, hasn't been visible before as clearly as in the example above. As long as the expressions were simpler or the data source was even a plain database query, its grouping feature looked like - well, grouping, and of course it can be seen and used like that. However, as soon as you understand the Generic Query as a transformation tool for turning data into the representation needed to understand and analyze it, you get a different - more abstract - view about it and its features and how they can be utilized and combined.

Having understood that the Generic Query is all about transforming data into an output representation, the first question you have to answer, after identifying the data, is, what type of output to generate:

  • columnar output (list output)
  • grid output (arranging data on a two-dimensional pane in rows and columns)

We did this distinction in the first two chapters of this guide, calling them linear queries and grouping queries.

Independent of that this decision is only used at the end of the processing and is also entered at the bottom of the input form, it determines how to mangle the data before submitting it to output generation. All the other features like Collection expression, Selection rules and Synthetic attributes, not mentioning the primary query itself, only serve to prepare the data for output generation.

If you see the Generic Query, instead of as a query tool (which it, of course, is), like this as an abstract engine providing abstract building blocks that can be combined in various ways, you will get a birds-view perspective on all query problems you will be faced with and will easily identify the patterns that need to be applied.

The cryptic role names and user IDs used in the output are not well suited to analyze and understand the data. Let's make the output more human-readable by adding the display names in both axes. As it turns out, this requires changing only the first two pipeline stages that fetch and aggregate the data for the first time. The aggregation keys used here are carried over into the output unmodified by all processing steps that follow. (Of course, when changing the aggregation keys we must ensure that this doesn't change the semantics of the aggregations. Adding the display name to the already unique name doesn't, and the same is true for the different sort order.)

So, what needs to be changed in the query's expression is only the count instruction and, as a consequence, the two additional attributes needed must be added to the attributes list in the lookup instruction fetching the data:

:Identity(>{name,displayName,assignedRoles.name,assignedRoles.displayName}\
  department == "DEP001" &&\
  assignedRoles.name.notNull()\
)\
:count(:format(${assignedRoles\.displayName,} (${assignedRoles\.name}))#:format(${displayName} (${name})))\
:group(value:map(key):sort,key)\
:flatten(#key)

ManagedAttribute → Application → Schema → displayAttribute

application.schema(type).displayAttribute

attributes:get(application.schema(type).displayAttribute)

A heavyweight example

Having done a lot of heavy lifting already, let's finish this guide with a last example to demonstrate one more advanced technique.

With version 8.3 IIQ introduced a new BundleProfileRelation class to ease examining the connections between entitlements and roles. The way these relationships are stored in IIQ using constraints in role profiles made them notoriously difficult to examine in the past, especially when coming from the entitlement side. The new table, maintained by a task that needs to be run regularly, is a great relief. Of course, where there is redundancy, there are always inconsistencies, so we are interested in checking how well the new mechanism represents the true data, which is still in the role profiles. Let's setup a query.

What we need to do is, for each role, to read the entitlements granted by the role the classical way from the profiles and the new way from the BundleProfileRelation table, and compare the results.

A straightforward way to do this would be to iterate over all roles, for each of them doing a query for the respective BundleProfileRelations and compare the result with what is derived from the role's profiles. The comparison can be easily done by either collecting the role's entitlements into a list using a bulletproof formatting like ${application.name}/${attribute}/${value} and sorting it, or more beautifully by building a nested mapping. We'll use the latter approach.

To get a better performance, instead of separate BundleProfileRelation queries for each role we will use a lookup table created from one big query.

Being unsure about if there is a referential integrity constraint between the BundleProfileRelation objects and the roles, the approach outlined so far might not report BundleProfileRelations for roles that don't exist any more. While we could live with that, we take the challenge to demonstrate a technique that allows to merge the results of two queries - the union instruction: We will collect the data to compare into two lookup tables, each mapping the roles to their entitlements. One will source the data from the profiles and the other one will source it from the BundleProfileRelations. Then we create the union key set of both lookup tables to iterate over all roles and for each role retrieve and compare the associated data from both lookup tables.

Let's first create the lookup tables. This code reads all roles and builds a mapping of role id → application name → attribute name → value. It iterates over all roles, their flattened required roles, their profiles, their constraints and the values they contain and counts the values by the mentioned keys:

:flatten(:Bundle(>!type.in({ "EntitlementRole", "OrganizationalRole" }))#flattenedRequirements#profiles#constraints#value)\
:count(^.^.^.^.id#^.^.application.name#^.property#@)

Building the same type of mapping from the BundleProfileRelation objects is a bit simpler:

:BundleProfileRelation(>required == true)\
:count(bundleId#sourceApplication.name#attribute#value)

Summing up and where to go from here

Having read up to here, you have seen many examples that illustrate the various ways how the building blocks offered by the Generic Query and the two DSLs can be combined to address different kinds of data exploration and presentation problems. The Generic Query can be applied not only to the obvious – identity and access management data – but also for examining the IIQ configuration, local files like log files, and basically anything that you can feed to it by writing a rule. This offers a way to use the data processing and presentation features of the Generic Query for any data to get it into a representation that answers specific questions.

By using the Generic Query from now on whenever you need to get insight into some of IIQ's data or other internals, you will get more and more familiar with its concepts and patterns, applying them automatically and finding more and more uses you would never have thought of, and at the end you will be able to work wonders with very little effort that you didn't think they are possible at all. Creating queries that better fit your personal workflows and habits than what you currently have will make you more productive and your work more fun. Stay healthy and enjoy!

This introduction has guided you through a quite remarkable journey, and you have now a good understanding of the Generic Query's functional principles and usage patterns. You are now ready to explore on your own the features that have not been covered here. Examine the reference documentation and the existing queries to discover more bells and whistles you can use. Here are some topics to dive into:

Reading input from rules

The Generic Query can read its input from a rule. Rules that support this are for instance six_sql_query, six_list_directory, and six_parse_file. This allows to apply the Generic Query's analysis and layout capabilities to any data you can write a rule for that returns this data in a collection or iterator.

By using six_beanshellrunner, you can even read from a Beanshell script embedded in your query! The following query groups the SIX AuditEventActionEnum values by the second and last word:

AuditAction enum matrix

Splitting large output into multiple tables

The Multiple* Grouping layouts provided by the Generic Query allow splitting the output of a single query into multiple tables or matrices, which are separated by captions. A navigator in the lower left corner of the output window allows to quickly navigate to any section.

See the Generic Query Reference for details.

Filtering the grouping result

This is similar to SQL's having clause: The Groups postprocessing field allows to filter the grouping result before it is processed for output. An example is reducing a huge matrix to the data of interest, for instance, keeping only cells with a value greater than some threshold, or rows containing data in at least so many columns.

Keeping detail in aggregated queries

When counting objects and presenting the result in a matrix, the nice thing is the overview this provides. What is uncomfortable is the effort it takes to get from the overview to the details to see what is behind the numbers. This is especially disturbing since all that information was already fetched by the query but then thrown away.

It is possible to preserve the detail information while still keeping the compact overview picture. The key to this is to, instead of only counting, collect the detail information by using the Output field or further groupings, and then use the Groups postprocessing to map the collected data to the number for display and attach the details to it in a tooltip (use the OrionQL link instruction).

The OrionQL and CerberusLibrary libraries provide prefabricated code you can use.

Rendering tables in popups and table cells

The OrionQL table, htable and grid instructions allow to explicitly render data in tabular and [multiple] matrix form. This allows to embed detail information into popups and table cells in a suitable format. Make yourself familiar with these instructions.

Using color

In highly aggregated queries, there is often not enough screen space to represent all desired dimensions of the data. An example would be a query that shows the ProvisioningTransaction counts for all endpoints for the last two weeks. To also show the success rate of the ProvisioningTransactions the numbers can be colored according to the success rate (and you want to also provide the detailed numbers in a popup as explained above). Make yourself familiar with the heat instruction and how to use the link instruction together with switch to color output.

The OrionQL and CerberusLibrary libraries provide prefabricated code you can use.

Creating charts

Numbers are nice, but to easily digest information, the human brain needs pictures. This is why charts are so popular. In addition, charts can also help to concisely represent additional dimensions of your data that would otherwise require more space.

OrionQL's two chart instructions allow to create charts with almost no effort – it was probably never so easy! Experiment with the barchart and linechart instructions.

Recursive flattening

The flatten instruction is, alongside the lookup instruction, the most powerful and demanding OrionQL instruction. It allows to navigate complex object models with ease.

What has not been covered in this guide is recursive flattening, that is, iterating over object chains as if they were collections. Be sure to read the reference and understand this powerful concept.

List and map decomposition

If you are familiar with languages like Python or Lua that support list decomposition by multiple assignment, you will immediately feel at home when seeing OrionQL code like this from the query in the above box:

List decomposition
subject,action=:match(six_([^_]+\).*?_([^_]+\)$)

The right side of the assignment extracts from a string composed of words connected by underscores the second and the last word, and these two values are assigned to the two variables on the left side.

What happens on a technical level is that the match instruction returns a list, and the multiple assignment decomposes this list into its elements, assigning them one-by-one to the specified variables (and discarding the unused elements).

This list decomposition is very handy. Use it, for instance, to split, with almost no code, a delimited string into its components – in this example an entitlement value made up as a colon delimited concatenation of mouleid, serviceid and accesslevel:

mouleid,serviceid,accesslevel=value:extract([^:]+)

Besides list decomposition, OrionQL assignments support also map decomposition. Map decomposition is requested by assigning a map to the special variable *. This has the effect that for each key contained in the map a variable with the corresponding value is created:

Map decomposition
*=attributes

Map decomposition can save you a lot of tedious and clumsy writing. Conditional formatting

When formatting text, there are often situations where missing pieces should be left out together with surrounding markup. Consider this formatting template:

${firstname} '${nickname,}' ${lastname}

It tries to account for the fact that nickname may be missing by providing an empty string as default. However, the two single quotes surrounding the placeholder will always be rendered. To avoid this, use a formatting template that contains the conditionally rendered markup (including one of the spaces):

${firstname} ${%'%s' ,nickname,}${lastname}

Now, if there is no nickname, lastname will directly follow firstname after a single space.

Memorizing data

The recache and swapcache instructions allow to use the caching mechanism to memorize data between iterations. Learn how this works – some queries need that.

Using macros

OrionQL and CerberusLogic macros allow to store reusable pieces of DSL code in macro libraries. The value of this in regard to reducing the complexity of queries, self-documentation and maintenance cannot be overestimated. Make yourself familiar with macros and the existing macro libraries (OrionLibrary, CerberusLibrary) and create a habit to watch out for code that can be reused – such as but not restricted to:

  • Object formatting
  • Data traversal
  • Standard output column sets

Making queries linkable

Learn how to store queries as Rule Runner Proxy Rules, so you can link to them from other, higher aggregated queries.

Composite expressions

OrionQL implements, beyond the expression alternative (|) that we used in this guide, a number of other binary operators (*, /, +, -, ==, !=, <, <=, >, >=), allowing as well to use parentheses for controlling the operator precedence. There is nothing unexpected in this area compared to other programming languages, so just remember that this is available for use. Maybe you want to have a special look at date arithmetic or multiplying strings though.

Computing aggregates

OrionQL provides four processing instructions to compute aggregates from collections (sum, avg, max, min). There is nothing unexpected here – just have a quick look and remember that these instructions exist when you need them.
