
# Rule Runner Authorization

Users with capability *SystemAdministrator* have unrestricted access to the Rule Runner.

Beyond that, rules are allowed to grant authorization by defining the following attributes:

- `authorizingCapabilities`  
  If this attribute is defined, it contains a comma separated list of capabilities that will grant access to the rule.
- `authorizingDynamicScopes`  
  If this attribute is defined, it contains a comma separated list of dynamic scopes (quick link populations) that will grant access to the rule.

Both approaches can be mixed. Example:

```xml
<Attributes>
    <Map>
        <entry key="authorizingCapabilities" value="WorkgroupAdministrator, WorkItemAdministrator"/>
        <entry key="authorizingDynamicScopes" value="Help Desk"/>
    </Map>
</Attributes>
```

Users with neither *SystemAdministrator* capability nor *any* of the capabilities declared by the rule and not belonging to any will be redirected to the *Access Denied* page when trying to invoke the respective rule through the Rule Runner.

> [!IMPORTANT]
> Proxy Rules
>
> When rules are invoked through Proxy Rules (see [Writing Rules for the Rule Runner](/spaces/OCA/pages/279282091/Writing+Rules+for+the+Rule+Runner)), their authorization is not checked. Instead, the authorization of the Proxy Rule applies.
